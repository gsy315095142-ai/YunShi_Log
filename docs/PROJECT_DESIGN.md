## 1. 项目结论

### 1.1 项目定位

一款**手机优先**的网页应用（响应式，手机浏览器可正常使用）。用户登录后管理个人信息、以日历形式记录每日事件，并可调用 AI「测算大师」对指定日期的记录进行解读。

后期部署至**阿里云服务器**长期运行，采用轻量技术栈，数据库使用 **SQLite**，避免过重的基础设施。

### 1.2 核心能力一览

| 模块 | 能力 |
|------|------|
| 账号 | 登录、开放注册；默认管理员账号；数据按用户隔离 |
| 个人信息 | 姓名、公历生日、可选出生时间、MBTI；自动推算农历、星座、五行、生肖 |
| 每日记录 | 日历视图；每天多条事件；格子摘要展示；增删改 |
| AI 测算 | 聊天窗口；发送某日记录给 AI；身份「测算大师」；支持 DeepSeek / 智谱 |
| 联网搜索 | 第一版预留接口；后续对接阿里云已部署的搜索工具 |

### 1.3 技术选型（已定）

| 层级 | 选型 |
|------|------|
| 前端 | 单页应用（SPA），手机优先布局 |
| 后端 | REST API |
| 数据库 | SQLite |
| 部署 | 阿里云单机常驻 |

> 具体框架（如 FastAPI + React/Vite）在开发阶段确定，须符合上述选型约束。

## 开发原则

开发时**禁止把全部功能堆进单一脚本**；按功能域拆分模块，功能点之间**尽量低耦合**，便于后期扩展与维护。

### 后端目录建议（按域拆分）

```
backend/
  app/
    main.py              # 仅应用入口、路由挂载
    auth/                # 登录、注册、JWT
    profile/             # 个人信息 CRUD
    records/             # 每日记录 CRUD
    fortune/             # 农历、星座、五行、生肖推算（纯计算，无 DB 依赖）
    ai/                  # AI 配置、聊天、厂商适配
      providers/         # deepseek.py、zhipu.py 各厂商独立
    search/              # 联网搜索对接（后续）
    db/                  # 连接、模型、迁移
```

### 前端目录建议（按页面/能力拆分）

```
frontend/src/
  pages/           # Login、Register、Profile、Daily、AI
  components/      # 可复用 UI（Calendar、ChatWindow 等）
  api/             # 按域拆分请求：auth.ts、profile.ts、records.ts、ai.ts
  hooks/           # 按页面或能力拆分自定义 Hook
  utils/           # 通用工具，不含业务逻辑
```

### 耦合控制要点

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个文件/模块只负责一件事（如 `fortune` 不算命、不读写用户表） |
| **接口隔离** | 模块间通过明确函数/接口调用，避免直接读对方内部变量或 SQL |
| **路由薄、服务厚** | HTTP 路由层只做参数校验与调用 service；业务逻辑放在 service 层 |
| **厂商可插拔** | DeepSeek、智谱各自独立 adapter，新增厂商只加文件、不改聊天主流程 |
| **推算与存储分离** | 命理推算为纯函数；profile 模块负责组数据，fortune 模块负责算 |
| **避免上帝文件** | 不出现「一个 routes.py / App.tsx 写完所有功能」的情况 |

## 2. 需求规格

### 2.1 账号系统

- **访问方式**：网页端，手机浏览器可正常访问。
- **登录**：账号 + 密码。
- **注册**：开放注册，任意用户可创建账号。
- **默认账号**：
  - 用户名：`Guosy`
  - 密码：`1234567890`
  - 角色：**管理员**（`admin`）
- **普通用户**：注册即为普通用户（`user`）。
- **权限**：第一版管理员与普通用户**功能无区别**，仅在数据模型中预留 `role` 字段，便于后期扩展。
- **数据隔离**：每个账号只能访问自己的个人信息、每日记录、AI 配置与聊天记录。

### 2.2 登录后主界面

三个页面，底部或顶部 Tab 切换：

| 顺序 | 页面 | 说明 |
|------|------|------|
| 1 | 个人信息 | 用户资料与命理相关展示 |
| 2 | **每日记录** | **默认打开页** |
| 3 | AI 测算 | 聊天式测算 |

### 2.3 个人信息页

#### 用户可输入字段

| 字段 | 必填 | 说明 |
|------|------|------|
| 姓名 | 是 | 文本 |
| 公历生日 | 是 | 日期，如 `1990-05-20` |
| 出生时间 | **否** | 时间，如 `14:30`；未填不影响其他推算 |
| MBTI | 否 | 如 `INTJ` |

#### 自动推算与展示字段（只读，由系统计算）

| 字段 | 计算依据 | 规则说明 |
|------|----------|----------|
| **星座** | 公历生日 | 按公历月日划分十二星座 |
| **天干五行** | 农历年 | 农历年天干对应五行（甲乙木、丙丁火、戊己土、庚辛金、壬癸水） |
| **纳音五行** | 农历年 | 农历年干支组合对应纳音（六十甲子两两一组，如己巳为大林木），显示完整名称 |
| **日主五行** | 公历生日 | 出生日日柱天干对应五行（如甲子日为甲木），八字本命核心 |
| **生肖** | 农历年 | 以**农历春节**为年界，非公历 1 月 1 日 |
| **农历** | 公历生日 | 自动补全展示，如「甲子年 三月 十五」；用户无需手填 |

> 三种五行格子各带 ⓘ 说明弹窗，解释算法差异（年柱 vs 日柱）。

#### 出生时间未填时的行为

- 农历展示到「日」即可。
- 星座、五行、生肖仍正常推算。
- 时辰相关字段显示「未填」或不展示。

#### 持久化

- 用户修改后保存至数据库，下次登录自动加载。

### 2.4 每日记录页

- **布局**：月历视图，可切换月份。
- **录入**：点击某一天，可新增该日的事件记录。
- **单条**：同一天仅允许一条记录（2026-07-20 变更，原为多条）；已有记录后只能编辑或删除，后端重复新增返回 409。
- **日历展示**：
  - 每个日期格子显示当日记录摘要（可取首条或合并简短预览）。
  - 内容过长时截断并显示**省略号**（`...`）。
- **编辑**：支持对已有记录进行修改、删除。
- **数据范围**：仅当前登录用户可见。

### 2.5 AI 测算页

- **交互**：类似聊天窗口，用户发送消息，AI 回复。
- **核心场景**：用户选择【每日记录】中**某一个日期**（及其记录内容），发送给 AI 进行测算解读。
- **AI 身份**：系统 Prompt 设定为「测算大师」。
- **个人档案注入**：每次对话自动附带用户命理档案（姓名、生日、农历、星座、生肖、天干五行、纳音五行、日主五行、MBTI），其中**日主五行标注为八字本命核心，作为主要分析依据**。
- **双层对话记忆**：
  - 最近 20 条消息原文携带；
  - 更早的消息滚动压缩为「历史对话摘要」（每积累 10 条旧消息触发一次 AI 压缩，增量并入摘要，存于 `ai_chat_summary` 表），随对话注入。
- **思考模式**：默认请求开启思考（智谱 `thinking.enabled`；DeepSeek 思考型模型自动返回 `reasoning_content`）。AI 回复的思考内容在独立「思考过程」框中展示，与发言气泡区分；未返回时显示"当前AI未返回思考内容"。
- **联网搜索**：已对接同机 SearXNG，搜索结果作为参考注入上下文。
- **厂商支持（第一版）**：

| 厂商 | API URL | 用户填写项 |
|------|---------|------------|
| DeepSeek | 自动填充默认 URL | API Key |
| 智谱 | 自动填充默认 URL（含套餐 URL 场景） | API Key |

- 用户选择厂商后，URL 由系统自动填入；用户主要只需填写 **API Key**。
- API Key 需安全存储（加密或环境隔离，不可明文暴露给前端）。

### 2.6 响应式与部署

- **布局**：以手机竖屏为主要适配目标；平板/桌面可正常使用但非首要优化对象。
- **部署**：阿里云服务器，服务常驻运行。
- **数据库**：SQLite，单文件、轻量、易备份。

## 3. 命理推算规则（实现参考）

### 3.1 公历 → 农历

- 用户只输入公历日期（+ 可选时间）。
- 使用成熟农历库换算，推荐：
  - **Python 后端**：`zhdate`、`lunardate`
  - **JavaScript 前端**（若前端即时预览）：`lunar-javascript`
- 建议由**后端统一计算**并返回，避免前后端规则不一致。

### 3.2 星座（公历）

按公历月日查表，示例边界：

| 星座 | 公历日期范围 |
|------|-------------|
| 白羊座 | 3/21 – 4/19 |
| 金牛座 | 4/20 – 5/20 |
| … | （按标准十二星座表实现） |

### 3.3 生肖（农历年）

- 以**农历春节**为年界。
- 例：1990-01-20 出生，公历已是 1990 年，但农历可能仍在己巳年（蛇）末，生肖应为**蛇**而非马。
- 实现：公历 → 农历日期 → 判断该日落在春节前还是后 → 取对应农历年的生肖。

### 3.4 五行（农历年）

- 根据农历年的**天干地支**确定年柱五行。
- 天干五行对应：甲乙木、丙丁火、戊己土、庚辛金、壬癸水。
- 展示时可只显示年柱五行（如「金」「木」），或展示天干地支全称（如「庚午」）。

## 6. API 设计（草案）

基础路径建议：`/api/v1`  
认证方式建议：**JWT**（`Authorization: Bearer <token>`）或 Session Cookie。

### 6.1 认证

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/auth/register` | 注册 `{ username, password }` |
| POST | `/auth/login` | 登录，返回 token |
| POST | `/auth/logout` | 登出（若用 Session） |
| GET | `/auth/me` | 当前用户信息含 role |

### 6.2 个人信息

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/profile` | 获取资料 + 计算字段（农历/星座/五行/生肖） |
| PUT | `/profile` | 更新 `{ display_name, birth_date, birth_time?, mbti? }` |

响应示例（GET）：

```json
{
  "display_name": "张三",
  "birth_date": "1990-05-20",
  "birth_time": "14:30",
  "mbti": "INTJ",
  "lunar": "庚午年 四月 廿七",
  "zodiac_sign": "金牛座",
  "chinese_zodiac": "马",
  "five_element": "金"
}
```

### 6.3 每日记录

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/records?year=2026&month=7` | 获取某月所有记录（按日期分组） |
| GET | `/records/{date}` | 获取某日全部记录 |
| POST | `/records` | 新增 `{ record_date, content }` |
| PUT | `/records/{id}` | 修改内容 |
| DELETE | `/records/{id}` | 删除 |

### 6.4 AI 配置

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/ai/settings` | 获取厂商、URL（Key 脱敏） |
| PUT | `/ai/settings` | 更新 `{ provider, api_key, api_base_url? }` |
| GET | `/ai/providers` | 返回支持的厂商及默认 URL 列表 |

### 6.5 AI 测算

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/ai/chat` | `{ message, linked_date? }` 发送测算请求 |
| GET | `/ai/chat/history` | 获取历史消息（分页） |

`POST /ai/chat` 服务端逻辑概要：

1. 读取最近 20 条历史消息（原文携带）；旧消息若积累满 10 条未压缩，先触发一次摘要压缩（失败不阻断聊天）。
2. 注入 system prompt（测算大师身份，日主五行为本命核心）。
3. 注入用户命理档案、历史对话摘要、（若带 `linked_date`）当日 `daily_records`。
4. 调用联网搜索（SearXNG）augment 上下文，失败自动跳过。
5. 调用用户配置的 DeepSeek / 智谱 API（默认请求思考模式），保存消息与思考内容并返回。

## 7. AI 厂商配置

### 7.1 第一版支持列表

| provider 值 | 显示名 | 默认 API Base URL |
|-------------|--------|-------------------|
| `deepseek` | DeepSeek | `https://api.deepseek.com` |
| `zhipu` | 智谱 | `https://open.bigmodel.cn/api/paas/v4` |

> 智谱若使用特定套餐，用户或系统可通过 `api_base_url` 覆盖默认值。

### 7.2 System Prompt（测算大师 · 草案）


## 9. 安全与运维

### 9.1 安全

- 密码 bcrypt 哈希存储，明文不落库。
- API Key 加密存储，接口返回脱敏（如 `sk-****abcd`）。
- 所有业务接口校验登录态；资源操作校验 `user_id` 归属。
- 生产环境 HTTPS；CORS 限制为前端域名。

### 9.2 SQLite 运维

- 数据库文件路径可配置，如 `data/app.db`。
- 定期备份 `.db` 文件。
- 写操作使用事务；注意并发写（单机部署通常足够）。

- 前端：构建静态文件，nginx 托管。
