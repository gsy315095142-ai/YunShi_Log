# Marketing Materials — 项目方案与结论纪要

> 本文汇总截至当前对话的结论，供新会话或协作时快速对齐范围与技术约定。若与代码实现不一致，以实现为准并应回写本文。

---

## 1. 产品目标（回顾）

构建 **Web + 本地后端** 的营销素材与小红书推文生成工具：

1. **素材管理**：在网页上传图片；支持**自定义分类**（新增 / 编辑 / 删除）。
2. **AI 配置**：用户填写 API Key；默认厂商与 API 地址可预填（历史讨论涉及 GLM、DeepSeek 等），具体默认值以实现为准。
3. **生成工作台**：对话或单轮需求输入；**主力 AI** 根据需求生成**小红书风格推文**，并结合需求**自动选用素材库中的图片**组合成可发布的文案方案。
4. **Skill 文档**：可添加与管理「技能」类文档，例如小红书推文范例、关键词表等；生成时注入上下文。

---

## 2. 部署与使用范围

| 项 | 结论 |
|----|------|
| **使用者** | 暂定**仅本人**使用。 |
| **认证** | **不需要登录**。打开本地网页即可使用。 |
| **安全假设** | 服务跑在本人机器/内网，认为只有本人可访问该站点。 |
| **后端** | **MVP 也需要后端**，可**部署在本地**；用于持久化 **API Key** 与 **上传的素材**（及元数据），避免密钥与文件仅依赖浏览器。 |

---

## 3. 数据存储

| 项 | 结论 |
|----|------|
| **数据库** | 使用 **MySQL**（作为正式、可迁移的生产级选择；本地开发可连接本机 MySQL 实例）。 |
| **素材文件** | 由后端存储（具体为本地目录或对象存储由实现决定，路径与元数据写入 MySQL）。 |

---

## 4. AI 架构（双角色）

采用 **两个逻辑上的 AI**，职责分离：

| 角色 | 职责 |
|------|------|
| **主力 AI** | 与用户「对话 / 理解需求」、生成小红书推文、编排内容与选用哪些素材（与识图结果配合）。 |
| **识图 AI** | **专门负责图像理解**（如对素材打标签、生成画面描述、辅助按内容选图等，以实现为准）。 |

**密钥约定**：

- 两个 AI **可以填写相同的 API Key**（例如同一平台下不同模型；或同一 Key 调用不同 endpoint/模型名）。
- 若使用不同平台，各自填写对应 Key 即可；产品层支持两套独立配置。

---

## 5. 模型名称与能力标注

在配置或界面上展示 **模型名称时，必须附带能力说明**，避免误选：

| 能力标签（示例） | 含义 |
|------------------|------|
| **纯文本** | 仅文本输入 / 输出，不能接收图片作为模型输入。 |
| **文本 + 识图（多模态）** | 可同时接收文本与图像，用于画面理解、多模态推理等。 |

**约定**：主力 AI 可选用纯文本或多模态（视产品策略）；**识图 AI 必须选用「文本 + 识图」类模型**。具体可选模型列表与默认项以实现为准。

---

## 6. 生成链路（原则）

- 生成内容须符合 **小红书** 常见结构（标题、分段、话题标签等；细节由 Skill 约束）。
- **配图须来自素材库**；若无合适素材，应明确提示补充，而非虚构图片地址。
- Skill 文档（范例、关键词等）在请求时**注入上下文**；注意上下文长度与截断策略（实现阶段细化）。
