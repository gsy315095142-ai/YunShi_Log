# Marketing Materials — 项目方案与结论纪要

> 本文汇总截至当前对话的结论，供新会话或协作时快速对齐范围与技术约定。若与代码实现不一致，以实现为准并应回写本文。

---

## 1. 产品目标（回顾）

构建 **Web + 本地后端** 的营销素材与小红书推文生成工具：

1. **素材管理**：在网页上传图片；支持**自定义分类**（新增 / 编辑 / 删除）。
2. **AI 配置**：用户填写 API Key 等；默认厂商与 API 地址可预填（历史讨论涉及 GLM、DeepSeek 等），具体默认值以实现为准。
3. **生成工作台**：对话或单轮需求输入；**主力 AI** 根据需求生成**小红书风格推文**，并结合需求**自动选用素材库中的图片**组合成可发布的文案方案。
4. **Skill 文档**：以多个 **Markdown 文件** 管理范例、关键词等；生成时按用户勾选注入上下文。

---

## 2. 部署与使用范围

| 项 | 结论 |
|----|------|
| **使用者** | 暂定**仅本人**使用。 |
| **认证** | **不需要登录**。打开本地网页即可使用。 |
| **安全假设** | 服务跑在本人机器，认为只有本人可访问该站点。 |
| **后端** | **MVP 也需要后端**，可**部署在本地**；用于持久化 **API Key（经配置文件）**、**结构化业务数据（JSON）**与 **素材元数据**；**图片文件**单独存放在用户可配置的目录中。 |
| **浏览器访问** | **先仅 `localhost`** 即可，不要求局域网访问。 |

## 3. 网络与监听

| 项 | 结论 |
|----|------|
| **监听地址** | 本地服务**暂时不必**监听 `0.0.0.0`（按常规本机访问即可）。若将来需要**局域网访问**，再增加该能力。 |

## 4. 数据存储

**原则**：结构化数据与配图文件分离——**JSON 仅存元数据与业务状态**，图片以文件形式落在磁盘指定根目录下。

| 项 | 结论 |
|----|------|
| **结构化存储** | 使用 **纯 JSON 文件**（可与图片存储目录分开；具体文件名与分拆粒度以实现为准，宜按模块或实体拆分以利于读写与备份）。**不使用 MySQL**（单机工具足够）。 |
| **素材文件** | **图片二进制**由后端写入用户配置的**素材根目录**；**路径、文件名、分类、备注等元数据**写入 JSON。 |
| **存储容量** | 可**向用户展示已占用空间**等信息，便于自行决策是否删除过期或不需要的素材（具体展示粒度以实现为准）。 |

**并发与安全写入**：单机单用户为主；实现上对同一 JSON 的写入应.serialize 一致或使用文件锁，避免异常中断损坏文件。

## 5. 素材与分类规则

| 项 | 结论 |
|----|------|
| **分类关系** | **一张图只对应一个分类**，**不允许多分类**。 |
| **默认名称** | 素材默认使用**图片上传时的文件名**；用户**可编辑**该名称。 |
| **备注 / 描述** | 可增加**备注**字段，作为**图片的文字描述**（供选图与生成参考）。 |

---

## 6. 图片存储路径

| 项 | 结论 |
|----|------|
| **默认值** | 由系统预设一个**默认素材根目录**（以实现为准，宜落在应用数据目录或用户文档下，避免与仓库源码混放）。 |
| **用户变更** | 用户可在界面中**修改素材根路径**并持久化；**新上传**写入新路径；已存在条目以元数据中记录的**实际文件路径**为准（首版可不强制一键迁移旧文件）。 |

---

## 7. 对话与历史记录

| 项 | 结论 |
|----|------|
| **历史保留** | 对话持久化至 JSON；**自动保留最近一段固定时长**，超出部分可删除。具体规则：**以会话「最后一条消息」的时间戳为准**，若距今超过 **7×24 小时**（滚动一周），则该会话可被自动清理（实现上可在后端启动或定时任务执行清理）。 |
| **用户操作** | 用户可 **自行删除** 某条/某会话历史、**跳转到之前的对话**继续查看或交互、或 **开启新对话**（与旧会话并列存在）。 |

## 8. AI 架构（双配置项）

主力 AI 与识图 AI 在产品中为 **两套独立配置项**，由用户自行填写（**厂商可以不同**）。每套配置可包含：Base URL、API Key、模型名等（字段以实现为准）。

| 角色 | 职责 |
|------|------|
| **主力 AI** | 与用户对话、理解需求、生成小红书推文、编排内容；在**工作流程中确需图像理解时**再触发识图能力。 |
| **识图 AI** | **专门负责图像理解**（画面描述、辅助选图等）；模型须具备**文本 + 识图（多模态）**能力。 |

**识图调用时机（编排策略）**：

- **按需识图**：**不在无关环节调用识图**；仅在**对话或工作流程中涉及图片识别**（例如需要理解素材画面、用户指明某图等）时，由主力 AI 通过后端提供的「工具 / 接口」触发识图 AI（具体协议在实现阶段定义）。
- **由主力 AI 决策**何时调用识图；系统提示中明确上述边界，避免无谓消耗。

**密钥**：

- 两套配置**可以**使用相同 API Key（同一厂商时），也**可以**使用不同厂商的不同 Key；以用户配置为准。

---

## 9. 模型名称与能力标注

在配置或界面上展示 **模型名称时，须附带能力说明**：

| 能力标签（示例） | 含义 |
|------------------|------|
| **纯文本** | 仅文本输入 / 输出，不能接收图片作为模型输入。 |
| **文本 + 识图（多模态）** | 可同时接收文本与图像。 |

**约定**：**识图 AI 必须**选用「文本 + 识图」类模型；主力 AI 以实现为准（可为纯文本，通过调用识图 AI 补全视觉信息）。可选模型列表与默认值以实现为准。

## 10. 密钥与 AI 配置的持久化

| 项 | 结论 |
|----|------|
| **存储方式** | API Key、默认/自定义 Base URL、素材根路径等写入**本地配置文件**（可与业务 JSON 分文件存放；格式以实现为准）。 |
| **前端交互** | 用户在前端录入时，输入框以 **\* 号等方式脱敏展示**；保存后写入配置文件（**不在前端长期明文展示完整密钥**）。 |
| **热更新** | **用户保存配置后应立即生效**：后续 API 调用与 AI 请求读取最新配置（实现可采用「每次读取」或「短 TTL 缓存 + 文件变更失效」）；不要求重启服务即可应用新配置。 |

| 项 | 结论 |

| **图片格式** | **允许各类常见图片格式**（具体解码与支持列表以实现为准，原则上尽量宽）。 |
| **单文件大小** | **暂时不做硬性上限**（占用本地空间；与第 4 节容量提示配合，由用户自行管理）。 |

## 12. Skill 文档

| 项 | 结论 |
|----|------|
| **目录** | Skill 文件统一放在项目（或运行时工作区）下的 **`skills/`** 目录中；系统按该路径扫描 `.md`。 |
| **形态** | 每个 Skill 为 **纯 Markdown（.md）文件**；可有**多个 Skill 文件**，**不必**合并为同一个 `.md`。 |
| **生成时选用** | 用户在生成前可 **勾选** 要纳入上下文的 Skill 文件；**默认全部勾选**，可取消勾选。 |
| **生效范围** | AI 在生成推文时，**仅使用被勾选的 Skill** 内容。 |
| **内容约束** | **生成内容须遵循已选 Skill** 中的规则、范例与关键词等（与主力 AI 的指令配合）。 |

**Skill 与配置热更新**：用户编辑 `skills/` 下文件或保存勾选状态后，**下一次生成请求应使用最新内容**（可对目录做按需读取或监听变更，以实现为准）。

---

## 13. 生成链路（原则）

- 推文风格与结构以 **已勾选的 Skill** 为主，并符合小红书常见形态（标题、分段、话题标签等）。
- **配图须来自素材库**；若无合适素材，应明确提示补充，而非虚构图片地址。
- Skill 较多时注意**上下文长度**与截断策略（实现阶段细化）。

## 15. 技术栈（无硬性偏好时的推荐）

以下为便于落地 AI 应用、生态成熟的**推荐组合**（实施时可据团队熟悉度微调）：

| 层级 | 推荐 | 说明 |
|------|------|------|
| **后端** | **Python + FastAPI** | 异步与 HTTP API 清晰；对接 OpenAI 兼容 / 各厂商 LLM 的客户端与示例多；读写本地 JSON、配置文件、按需代理识图调用都较直观。 |
| **前端** | **React + TypeScript + Vite** | 组件生态与 UI 库丰富；类型系统利于维护配置表单、Skill 勾选与对话界面。 |
| **备选** | Node.js（如 **NestJS** 或 **Hono**）+ **React + TS** | 全栈 TypeScript，前后语言统一。 |

**持久化**：本地 JSON 文件 + 磁盘图片。**访问范围**：先 localhost。

## 17. 开发计划（阶段）

以下为推荐实施顺序；各阶段验收后再展开下一阶段，可依进度并行前后端边界清晰的任务。

| 阶段 | 内容 | 主要产出 |
|------|------|----------|
| **P0 — 脚手架与数据底座** | 初始化前端（React+Vite+TS）与后端（FastAPI）；定义 `skills/`、`data/`（或等价）与配置文件 layout；默认素材目录与写入安全策略；静态资源路由（若前端由后端托管或由 Vite 代理联调）。 | 可启动的前后端、`skills/` 可读、配置文件可写。 |
| **P1 — 配置与 Skill** | 双 AI 配置表单（脱敏）；保存至本地文件并实现**保存即生效**；Skill 列表 API + 勾选状态持久化；生成请求拼接已选 Skill 文本。 | 配置页、Skill 勾选、后端读配置无缓存陈旧问题。 |
| **P2 — 素材模块** | 分类 CRUD；上传/列表/编辑名称与备注；**可变更素材根路径**及元 JSON 一致；占位或基础容量统计。 | 完整素材管理能力。 |
| **P3 — 对话工作台** | 会话与消息 CRUD JSON 模型；新会话 / 切换 / 删除；**滚动 7×24 小时**清理逻辑；前端对话 UI；主力 AI 流式或非流式对话接口。 | 可用对话工作台 + 自动清理策略。 |
| **P4 — 生成链路** | 小红书风格 system 提示；约束「配图仅素材库」；上下文长度与 Skill 截断策略；端到端生成与展示。 | 从需求到成稿闭环。 |
| **P5 — 识图编排** | 后端 tool/内部接口：**仅在有图相关需求时**由主模型触发识图；识图结果回注对话；超时与错误提示。 | 按需识图、不空跑多模态。 |
| **P6 — 打磨** | 错误处理、加载态、纯中文文案统一；可选：导出对话、简单备份说明（拷 JSON + 图片目录）。 | MVP 可日常自用。 |
